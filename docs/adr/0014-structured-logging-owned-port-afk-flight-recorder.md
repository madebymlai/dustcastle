# Structured logging behind an owned `Logger` port, on by default, with a persisted byte-bounded flight recorder for AFK runs

## Status

accepted — replaces the ad-hoc `onLine: (line: string) => void` seam that today threads through ~40 functions across `store/`, `sandbox/`, `run/`, and `cli/` (`AutoGcOptions.onLine`, `EnsureEgressOptions.onLine`, `EnsureImageOptions.onLine`, `PrepareOptions.onLine`, each defaulted to `?? (() => {})`). Builds on [ADR 0007](0007-store-lifecycle-management.md) (the post-run `__autogc` sweep and "never worry about disk" byte-ceiling ethos) and the rootless per-user `DUSTCASTLE_HOME` ownership it already exercises (`recency.json`, `gc.lock`, `gc.log`).

## Context

dustcastle already has a logging **port** — every deep module accepts an injected `onLine` sink and defaults to silent; only `cli/*` binds a real sink (`(l) => process.stderr.write(\`${l}\n\`)`). But it is a bare `(string) => void`: no levels (severity is faked with `"gc: WARNING …"` string prefixes), no structured fields, and no capture — the lines stream to a terminal and are gone.

That last gap is the one that matters, because **dustcastle runs AFK.** You start `dustcastle run`, watch the posture banner, and walk away while the plan→implement→review→merge loop grinds over the repo's beads issues unattended. The expensive failure mode is not a noisy terminal (nobody is watching) — it is *coming back to a broken run with nothing to reconstruct what happened.* A quiet-by-default CLI is exactly wrong for an unattended tool.

dustcastle is also **both a library and an application**: it is published to npm (`"private": false`, has a `bin`) and its deep modules are importable, while `cli/*` is the composition root. The prevailing ecosystem guidance ([`abstract-logging`](https://github.com/jsumners/abstract-logging), by a pino-core maintainer; LogTape's library-vs-app comparison) is explicit: library code must **not** hard-depend on a concrete logger like pino — it should accept an injected logger and fall back to a noop — while the application binds the concrete logger at its composition root. dustcastle's existing `onLine ?? noop` shape is already that pattern, informally.

## Decision

**Upgrade the `onLine` seam to a leveled, structured `Logger` port owned by dustcastle, bound to pino only at the CLI, with output on by default to two asymmetric sinks — a human stderr stream and a persisted full-fidelity flight recorder that the existing post-run sweep prunes to a byte budget.**

- **Own the port; pino is one adapter.** A dustcastle-owned `interface Logger` (the six standard levels `fatal/error/warn/info/debug/trace` + `child(bindings)`) lives in a new `src/log/` deep module alongside `noopLogger` (the new default, replacing `?? (() => {})`). The port's call signature is **pino-native** — `info(mergingObject?, msg?, ...args)` — so pino is a **structural drop-in** with no translation layer, yet deep modules `import type { Logger }` from `src/log/`, never from `pino`. The `pino` import is contained to **adapter edges only** — `src/log/pino.ts`, the host `createLogger()` adapter — never in a deep module or the `Logger` port. (Refined 2026-06-05: the egress proxy adds a *second*, deliberate edge — `src/sandbox/proxy-logger.ts` — at its process boundary; see the amendment below. The invariant is "only at the edges," not "exactly one file.") This is the same "external mechanism behind a dustcastle seam" discipline the codebase already applies to the sandcastle `Provider`, the `Pool` brain, and `ImageSpec` — pino is "an implementation detail, not dustcastle's identity."

- **Structured, not stringly.** Call sites pass fields as data (`log.info({ freedBytes, storePath }, "collected")`) rather than baking everything into the message. The subsystem prefix that today lives in the string (`"gc:"`, `"egress:"`, the `EnsureImageOptions.logPrefix` field) becomes a `mod` **binding on a child logger**, so the flight recorder is sliceable (`jq 'select(.mod=="egress")'`). **The caller names the child** (`logger.child({ mod })` at the composition root); deep modules stay label-agnostic — they log to whatever `Logger` they are handed.

- **On by default — because AFK.** The logger is not silent-by-default. The whole point is that diagnostics are *there* when you return to a broken unattended run, without having had to pass `--verbose` before walking away (which you couldn't, because you didn't know it would break).

- **Two asymmetric sinks (pino multi-target transport).**
  - **stderr** — rendered by **pino-pretty** (legible for the watched startup and `jq`-free eyeballing), default level `info`, tunable via `DUSTCASTLE_LOG=debug|info|silent`.
  - **`DUSTCASTLE_HOME/runs/<timestamp>.jsonl`** — the **flight recorder**: raw JSON, always at `trace` (full fidelity — every nix invocation, store path, lockfile hash, egress decision, timing), **untunable**. stderr is the dashboard you can quiet; the file is the black box that always records. `DUSTCASTLE_LOG=silent` mutes stderr for a pure-AFK run while the recorder keeps writing.

- **Retention is a byte ceiling, pruned by the existing sweep.** `runs/` is bounded by a byte ceiling (evict oldest first), pruned inside the detached post-run `__autogc` sweep that ADR 0007 already spawns — no new lifecycle, no new daemon. The flight recorder inherits the Store's "tend it for you" guarantee. It is deliberately **not** a `Pool`: logs are write-once history, never re-warmed, so the recency/ceiling brain (`collectPool`) does not apply — a dumb byte-bounded prune is correct. `src/log/` owns this prune (its own artifact, end-to-end), not `store/`.

  - **The ceiling is a flat 16 MiB (2²⁴ = 16,777,216 bytes), a deliberate deviation from ADR 0007's "no absolute numbers" rule.** ADR 0007 derives every Store threshold as a *fraction of disk* because a content-addressed package store should grow with the machine. Run logs are the opposite kind of artifact: a run's log is ~the same size on any machine (~tens-to-hundreds of KB of dustcastle's own `trace` events, not the agent transcript), and its value is **recency** — you only ever read the last broken run, never the 200th-oldest. So it is *recency-bounded, not capacity-bounded*: scaling it to disk (hundreds of MB on a 4 TB box) would hoard stale logs nobody reads for no benefit. 16 MiB holds ~50 typical runs (and ~16 even for long, chatty AFK sessions) — comfortably past the recency window — while staying a rounding error against the Store. The flat cap is the *more* correct model here precisely because logs do not scale with disk.

  - **One sweep process, two cleanup competencies.** The post-run `__autogc` sweep is a single detached spawn under a single lock with a single report. Inside it, two *different* mechanisms now run: the pool-GC brain (`collectPools`) over the reusable warm/cold artifacts (Store + deps cache), and this flat byte-prune over the flight recorder. They share the trigger, lock, and report but not the logic — because the flight recorder is not a Pool. This asymmetry is intentional, not a smell: caches get the recency/ceiling brain; write-once logs get a dumb oldest-first byte-prune.

- **One output path (the banner included).** The posture banner (`printPosture`'s `🏖️`) and the `🧹` sweep line stop being `console.error` blocks and become **structured events** (`{ event: "provisioned", ecosystems, mode, egress, toolchains }`, `{ event: "swept", … }`). The stderr transport's `messageFormat(log)` special-cases these events to render the multi-line banner / line forms; everything else renders as `mod: msg`. Consequence: the banner is now *in* the flight recorder as queryable data. The **one exception** is `src/cli/select.ts` — the interactive model picker is raw-mode TTY *input*, not an event stream, and stays as-is.

- **The egress proxy gets pino too, but no flight recorder.** The proxy process (`proxy-main.ts`) is a separate process in a separate container and cannot reach `DUSTCASTLE_HOME`. It binds pino — through its own adapter, `src/sandbox/proxy-logger.ts`'s `createProxyLogger()` — to **container stderr** (JSON, captured by the runtime) using the same `Logger` port, emitting structured events (`{ event: "listening", port }`, `{ decision: "deny", host }`). **Companion change:** the host-side liveness probe `PROXY_LISTENING = /dustcastle-egress: listening on/` in `egress-runtime.ts:50` (and its test fixture) must be rewritten to parse the JSON `event: "listening"` — otherwise provisioning hangs waiting for a readiness string that the proxy no longer prints. This makes the proxy↔host readiness signal a structured contract, not a stderr-prefix grep.

## Considered Options

- **Keep `onLine`, add only a level enum.** A ~15-line hand-rolled leveled logger, no dependency. Rejected: it gives levels but not capture, and the AFK flight-recorder requirement (persisted, byte-bounded, multi-sink, full-fidelity-file/curated-stderr) is exactly what pino's multi-target transport does for free. The dependency earns its keep.
- **Use pino's type as the interface (`logger: pino.Logger` on every options object).** Less indirection, full native pino API everywhere. Rejected: it couples `store/`, `sandbox/`, `run/` — library code — directly to a logging vendor, against both the house ports-and-adapters style and the library-vs-app guidance. The owned port costs almost nothing (pino satisfies it structurally) and keeps the vendor swappable and tests pino-free.
- **Message-first signature (`info(msg, fields?)`).** Reads more naturally at call sites. Rejected: it forfeits pino's zero-adapter structural compatibility for ergonomics and re-introduces an arg-flipping wrapper to test; the fields-first idiom is well-known pino.
- **Silent / quiet by default.** The conventional CLI default. Rejected: wrong for an AFK tool — it optimizes for a human watching the terminal, which is precisely the case that does not hold here.
- **stderr only, no persisted file.** Simplest; relies on the user redirecting output before walking away. Rejected: "use it how it's intended — walk away" means the user did *not* set up redirection, so "it's there if you need it" requires persistence.
- **Same level on both sinks.** Simpler. Rejected: if stderr defaults to `info`, the `debug` detail needed for a post-mortem was never recorded — which defeats having a file at all. The asymmetry (file `trace`, stderr `info`) is the point.
- **Count-based retention (keep last N runs).** Simpler prune. Rejected in favor of a byte ceiling to match ADR 0007's existing "measure bytes, evict to budget" disk guarantee exactly.
- **Module name the child loggers / module self-names (`store/gc.ts` does `.child({ mod: "gc" })`).** Matches today's "each module owns its prefix" convention. Rejected: caller-naming keeps deep modules fully label-agnostic (purer DI), and preserving the old convention is not a requirement.
- **Proxy out of scope (keep its `dustcastle-egress:` stderr prefix).** No blast radius on the proxy↔host contract. Rejected: structured egress decisions are valuable for debugging denials, and the liveness probe is a small, contained rewrite.

## Consequences

- **Migration is a triage, not a rename.** Each of the ~40 `onLine` sites becomes a leveled, structured call: curated progress beats → `info` (visible on stderr), mechanism detail (nix invocations, store paths, lockfile hashes, timings) → `debug` (file-only), the `"WARNING"`-prefixed lines → `warn`, failures → `error`. The `onLine` field leaves every options interface; `EnsureImageOptions.logPrefix` is absorbed into a child binding.
- **The proxy↔host readiness contract changes.** `egress-runtime`'s liveness probe and its test move from string-prefix matching to JSON parsing. This is the one cross-process blast-radius item and must land with the proxy change, not after.
- **`pino` (+ `pino-pretty`) become runtime dependencies**, imported only at the edges (`src/log/pino.ts` for the host, `src/sandbox/proxy-logger.ts` for the proxy). The library half stays vendor-neutral.
- **Tests get a trivial in-memory `Logger`** (an array-collecting fake satisfying the six methods + `child`) — no pino, no transport teardown in unit tests.
- **The banner is no longer hand-printed** — it is one structured event with a custom renderer, captured in the flight recorder. `printPosture` becomes an emit + a `messageFormat` branch.
- **CONTEXT.md gets nothing.** A `Logger`/flight recorder is infrastructure, not domain vocabulary; the glossary (Store, Sandbox, Toolchain, Egress, Deps Cache) stays a pure domain glossary.
- **Future enhancement (out of scope):** bridging proxy decisions into the *host* flight recorder (have `egress-runtime` parse the proxy's JSON events and re-emit them through the host logger) — deferred; the proxy currently logs only to its own container stderr.

## Amendment (2026-06-05): pino lives at adapter *edges*, not "exactly one file" — the proxy is a sanctioned second adapter

The Decision's headline invariant originally read *"the `pino` import is contained to exactly one file (`src/log/pino.ts`)."* That was imprecise and contradicted this same ADR: the proxy clause and the Consequences above both describe a **second** pino binding for the egress proxy. The load-bearing invariant is **pino imported only at adapter edges — never in a deep module or the `Logger` port** (which is what keeps the library half vendor-neutral). There are two such edges, and that is correct:

- `src/log/pino.ts` — `createLogger()`, the host adapter: multi-target transport (pino-pretty stderr + the `runs/<ts>.jsonl` flight recorder), stderr level `DUSTCASTLE_LOG`-tunable.
- `src/sandbox/proxy-logger.ts` — `createProxyLogger()`, the proxy adapter: raw JSON to container stderr only, level pinned `trace`, no run file. Factored out of `proxy-main.ts` for testability — `proxy-main.ts` only *calls* it. (The original Decision/Consequences named `proxy-main.ts` as the edge; the binding has since moved here.)

**The proxy is a deliberate second adapter at a process boundary, not a seam leak.** It cannot reuse the host adapter, for three load-bearing reasons:

1. **Homeless.** The proxy is its own container (ADR 0011) with no `DUSTCASTLE_HOME`; `createLogger` unconditionally `mkdirSync`s the run-log dir and writes a host-path file the proxy cannot reach.
2. **Its stderr is a machine contract, not a human dashboard.** `ensureEgress` proves the proxy is *serving* (never a silent success — ADR 0011) by `JSON.parse`-ing the container's log lines and matching `event: "listening"` (`egress-runtime.ts`). The host adapter renders stderr through pino-pretty, whose `ignore` list strips the `event` field — so reusing it would break the liveness probe and hang provisioning.
3. **Different level policy.** The proxy's stderr is its *only* sink (pinned `trace`); the host's is a `DUSTCASTLE_LOG`-tunable secondary behind the always-`trace` file.

By the house seam test (*one adapter = hypothetical seam; two = real*), the proxy is what makes the `Logger` port earn its keep. **Do not re-flag the two `pino` imports as a binding leak, and do not collapse `createProxyLogger` into `createLogger`** — merging would force the homeless proxy to carry `homeDir`/retention params it cannot honor and would re-break the liveness contract.
